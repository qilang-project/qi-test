# qi-test

用 **Qi 写的**测试框架，风格对标 `go test` / `cargo test`：原生 `qi test` 命令发现并运行测试、
命名用例 + 逐用例计时、`stub/spy`、假 LLM 服务器，失败非零退出（进 CI）。

给整个 qilang 生态（qi-harness / qi-web / qi-cli …）做 `.qi` 级确定性测试。
跟 `qi/scripts/run_examples.sh`（示例回归）互补：那个验证「示例能跑」，qi-test 验证「逻辑正确」。

## 结构

```
qi-test/
├── qi.toml          # 包配置（包名「测试」）
├── 测试.qi          # 框架：运行用例 / 摘要并退出 + 断言库；re-export 桩
├── 桩.qi            # stub / spy（测试替身，固定池 8 槽）
├── qi_packages/
│   ├── Harness -> ../../qi-harness   # 被测包（需 qi-harness 同级存在）
│   └── Web     -> ../../qi-web       # 假模型服务器用
├── 用例/                              # 单元（纯本地，无网络）
│   ├── 并发派发_测.qi    # 工具::并发派发（结果对齐 + 并发计时）
│   ├── 结构化_测.qi      # 校验JSON对象（结构化输出原语）
│   ├── 文件沙箱_测.qi    # 文件工具 沙箱/权限安全（自带 symlink fixture）
│   └── 桩_测.qi          # stub/spy 自测
├── 工具/
│   └── 假模型服务器.qi   # OpenAI 兼容的「假 LLM」（qi-web），返回脚本化回合
├── 集成/                              # 集成（跨进程，要起服务器）
│   └── 假模型_集成测.qi  # 假模型驱动 agent 工具循环，端到端确定性测
└── 跑集成.sh        # 集成编排：起假模型服务器 + 跑 集成/*（见下「为何还有一个 sh」）
```

## 跑测试

```bash
# 前提：先 cargo build（产出 qi 编译器），且 qi-harness/qi-web 在同级目录
qi test 用例              # 跑 用例/ 下所有 *_测.qi（递归发现），聚合 + CI 退出码
qi test                  # 默认当前目录递归找 *_测.qi
qi test 用例/桩_测.qi     # 跑单个文件
qi test 用例 -f 桩        # 只跑名字含「桩」的用例（≈ go test -run）
qi test 用例 -v          # 详细：连通过的断言也打印
```

`qi test` 是编译器内置子命令（`qi/src/cli`）：递归发现 `*_测.qi` → 逐个编译+运行 →
按各文件退出码判套件成败 → 打总报告 → 有失败整体 `exit 1`。**无需 shell 脚本。**

输出（cargo test 风格）：

```
▶ 文件沙箱_测
测试 文件沙箱.拒越界路径 ... 通过 (3ms)
测试 文件沙箱.拒符号链接逃逸 ... 通过 (1ms)
...
测试结果: 4 通过 / 0 失败，共 4 个测试（断言 10/0）
结果: ok
  ✓ 套件通过

════════════════════════════
测试套件: 4/4 通过
```

## 写用例

测试是 `函数(): 整数`（约定 `返回 0`），用 `运行用例(名字, 函数)` 跑；每个用例独立计时、
独立判定（一个用例的断言失败不影响别的）：

```qi
包 主程序;
导入 测试::{运行用例, 摘要并退出, 断言相等整数, 断言包含};

函数 测_加法() : 整数 {
    断言相等整数(1 + 1, 2, "1+1=2");
    返回 0;
}
函数 测_子串() : 整数 {
    断言包含("hello world", "world", "含 world");
    返回 0;
}
函数 入口() {
    运行用例("算术.加法", 测_加法);
    运行用例("字符串.子串", 测_子串);
    摘要并退出();         // 打总账 + 按结果 exit 0/1
}
```

文件名以 `_测.qi` 结尾即被 `qi test` 发现。

### 断言 API（`包 测试`）

```
断言为真(条件整数, 说明)            断言为假(条件整数, 说明)
断言相等整数(实际, 期望, 说明)       断言相等(实际串, 期望串, 说明)
断言包含(文本, 子串, 说明)          断言不含(文本, 子串, 说明)
运行用例(名字, 函数(): 整数)        摘要并退出()
```

默认「过了不吭声、错了才打」（go test 默认）；`-v` / `QI_TEST_VERBOSE=1` 连过的也打。
`-f 子串` / `QI_TEST_FILTER=子串` 只跑名字含该子串的用例。

## stub / spy（测试替身）

Qi 没闭包，工具处理器签名固定 `函数(字符串):字符串`，无法捕获「我是哪个桩」，所以用固定池（8 槽）：

```qi
导入 测试::{新桩, 桩设返回, 桩取处理, 桩调用次数, 桩末次参数, 桩重置};

变量 s: 整数 = 新桩();
桩设返回(s, "{\"temp\":18}");                  // stub：固定返回值
变量 h: 函数(字符串):字符串 = 桩取处理(s);
代理值 = 添加工具(代理值, (工具 { 名字:"查天气", 处理: h, … }));
// …跑被测逻辑…
断言相等整数(桩调用次数(s), 1, "查天气被调一次");   // spy：调用次数
断言包含(桩末次参数(s), "Tokyo", "末次参数对");     // spy：末次参数
```

## 假 LLM 服务器（集成测试）

真 LLM 走 FFI 打 HTTP，没法单进程 mock。`工具/假模型服务器.qi` 是个 OpenAI 兼容的 qi-web 假服务器，
返回**脚本化回合**（环境变量 `QI_FAKE_SCRIPT`，每行一回合）：

```
工具|<工具名>|<参数JSON>     → 让模型「调用某工具」
文本|<答复文本>              → 让模型「给最终答复」
```

把 `QI_LLM_BASE` 指到它，就能确定性地测整个 agent 工具循环。`集成/假模型_集成测.qi` 用桩当工具，
验证「模型 round1 发 tool_call → agent 派发 → round2 给最终答复」。

```bash
./跑集成.sh        # 起服务器 + 设脚本 + 跑 集成/* + 收尾
```

### 为何集成还留着一个 sh？

单元测试已全部由原生 `qi test` 跑，**没有 shell**。集成测试要管理一个真 HTTP 服务器进程的
**生命周期**（启动 → 探活 → 收尾 + 端口清理），这正是 shell 擅长、且能避开「杀父进程留孤儿服务器占端口」
的活儿 —— 跟 `go test` 里集成测试常配 TestMain/外部脚本起服务同理。把它做进 Qi 也行（`子进程` + `标准库.HTTP` 轮询），
属于后续可选项。
